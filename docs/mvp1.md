# LEXI — MVP Section 1 개발 회고

> "배운 개념이 한 학기 뒤 사라지는 문제"를 해결하기 위해 만든 AI 지식 사전. Section 1에서 핵심 플로우 전체를 구현했다.

---

## 1. 왜 만들었나

AI 논문을 읽거나 강의를 들을 때, '감각'은 남는다. 하지만 언어로 꺼내려 하면 이미 사라져 있다.

기존 도구들의 문제는 명확했다.

| 도구 | 문제 |
|------|------|
| Notion | 완성된 노트를 만들어야 한다는 부담 |
| Obsidian | 링크 설정 피로 |
| NotebookLM | 노트북 간 연결 없음 |
| Recall | 키워드 연결만 있고 '내 맥락'이 없음 |

공통 문제는 하나였다. **"내가 왜 이 개념에 꽂혔는지"를 저장하지 않는다.** 연결의 기준이 키워드가 아니라 내 이해의 맥락이어야 한다고 생각했다.

그래서 LEXI를 만들기 시작했다.

---

## 2. 솔루션 컨셉

> PDF를 던지면 → 내 말로 개념을 정착시키고 → AI와 짧은 대화로 깊이를 조절하고 → 자동으로 연결이 쌓이는 나만의 지식 사전.

핵심 차별점 세 가지:
- 연결의 기준 = 키워드/태그가 아닌 **내가 이해한 맥락**
- 입력 마찰 최소화 — PDF 올리고 한 줄만 써도 저장 완료
- AI가 연결 제안 시 **왜 연결됐는지 이유 포함**

---

## 3. 기술 스택

| 영역 | 기술 | 선택 이유 |
|------|------|-----------|
| Frontend | React 19 + Vite + Tailwind CSS | 빠른 개발 속도, 유틸리티 우선 스타일링 |
| Database | Firebase Firestore | 프론트 직접 호출 가능, 무료 티어 충분 |
| AI | Gemini API (gemini-2.0-flash) | PDF 멀티모달 처리 + 빠른 응답 |
| 그래프 시각화 | react-force-graph-2d | 포스-다이렉티드 그래프, 가벼운 API |
| 배포 | Vercel | 정적 빌드 배포 |

### 스택 전환 결정 — Supabase → Firebase

원래 계획은 Supabase(PostgreSQL)였지만, 개발 시작 시점에 무료 플랜 한도가 이미 소진된 상태였다. 대안을 검토했다.

- **Firebase Firestore 선택 이유**: 백엔드 서버 없이 프론트에서 직접 호출 가능하다는 아키텍처 원칙을 유지할 수 있었고, 단일 사용자 토이 프로젝트 기준으로 무료 티어(읽기 50K/일, 쓰기 20K/일)가 충분했다. 관계형 쿼리가 없어도 되는 데이터 구조였기 때문에 NoSQL 전환 비용도 낮았다.

## 4. 구현한 기능 (Section 1 MVP)

### 4-1. PDF 업로드 & 개념 추출

PDF 파일을 드래그앤드롭으로 올리면, FileReader로 base64 인코딩 후 Gemini API의 멀티모달 기능으로 핵심 개념을 추출한다. 결과는 `[{ name, description }]` JSON 배열로 받아 사용자가 직접 편집(추가/수정/삭제)할 수 있다.

```
PDF → base64 인코딩 → Gemini (멀티모달) → 개념 JSON → 사용자 편집
```

**고민했던 점**: Gemini에게 자유 형식 텍스트 대신 JSON 배열만 반환하도록 프롬프트를 구성했다. 응답에서 정규식으로 JSON만 파싱하는 방어 로직을 추가해 LLM의 불안정한 응답 형식에 대응했다.

### 4-2. 인사이트 입력 & 학습 대화

단순한 노트 저장이 아니라 '내가 왜 이 개념이 대박이었나'를 한 줄로 쓰면, Gemini가 이를 읽고 학습 방향 버튼 3개를 제안한다.

```
더 깊이 파고들기 / 다른 개념과 연결해보기 / 지금 바로 저장
```

선택에 따라 최대 5턴의 학습 대화가 이어진다. 대화 맥락을 유지하기 위해 히스토리를 두 가지로 분리해서 관리했다.

- **displayHistory**: UI에 표시되는 대화 기록
- **apiHistory**: Gemini에 전송하는 전체 히스토리 (시스템 컨텍스트 포함)

이렇게 분리하면 사용자는 깔끔한 채팅 UI를 보고, Gemini는 카드 제목·개념·인사이트 등 풍부한 컨텍스트를 받는다.

### 4-3. Firestore 저장 & 연결 제안

저장 흐름에서 한 가지 순서 문제를 신경 썼다. 새 카드를 저장하기 **전에** 기존 카드 목록을 먼저 가져온다. 저장 후에 가져오면 새 카드가 자기 자신과 연결 후보로 잡히기 때문이다.

```
기존 카드 조회 → 새 카드 저장 → Gemini 연결 제안 → 사용자 선택 → connections 저장
```

Gemini에게 새 카드와 기존 카드 목록을 함께 넘겨 의미 있는 연결을 제안받는다. 각 제안에는 **연결 이유 한 줄**이 포함되어, 사용자가 맥락을 확인하고 선택할 수 있다.

### 4-4. 카드 상세 페이지

저장된 카드를 상세하게 볼 수 있다. 핵심 개념 태그에 hover하면 설명 툴팁이 뜨고, 인사이트는 블록쿼트로 강조 표시된다. 학습 대화 기록은 ChatBubble 컴포넌트를 재사용해 그대로 표시된다.

연결된 카드 목록은 양방향 연결(card_id_a ↔ card_id_b)을 모두 쿼리해서 표시하고, 각 연결에 AI가 생성한 이유가 함께 보인다.

### 4-5. 그래프 탐색 페이지

전체 카드와 연결 관계를 react-force-graph-2d로 시각화했다. 왼쪽은 검색 가능한 카드 목록, 오른쪽은 포스-다이렉티드 그래프다.

- 노드 클릭 → 카드 상세로 이동
- 링크 hover → 연결 이유 툴팁
- 파티클 애니메이션으로 연결 방향 표시

## 5. 아키텍처 결정사항

### 백엔드 없는 구조

Supabase든 Firebase든, 이 프로젝트의 핵심 원칙은 **백엔드 서버 없이 프론트에서 직접 API 호출**이었다. Gemini API도 SDK 없이 직접 `fetch`로 호출한다. 덕분에 인프라 복잡도가 없고 Vercel 정적 배포만으로 끝난다.

### 단일 페이지 상태 머신 (UploadPage)

Upload 플로우가 5단계(upload → extracting → extract → insight → chat → connect)에 걸쳐 있어서, 하나의 컴포넌트 내 `step` 상태로 전체를 관리했다. 각 단계는 별도 서브 컴포넌트 함수로 분리해 가독성을 유지했다.

### Gemini 응답 파싱 방어 로직

LLM 응답은 형식이 일정하지 않다. 모든 Gemini 호출에서 JSON이 필요한 경우, 정규식으로 JSON 블록만 추출하고 파싱 실패 시 빈 배열을 반환하는 패턴을 일관되게 적용했다.

```javascript
const jsonMatch = raw.match(/\[[\s\S]*\]/)
if (!jsonMatch) return []
return JSON.parse(jsonMatch[0])
```

└── App.jsx             # BrowserRouter + 3개 라우트

## 7. Section 2 고려사항

MVP를 쓰면서 자연스럽게 생긴 아이디어들. 지금은 건드리지 않는다.

- **카드 삭제/편집**: 현재 저장 후 수정 불가. Card 페이지에 편집 기능 추가 필요
- **복습 알림**: 저장한 카드를 며칠 후 다시 떠올리게 하는 스페이스드 리피티션
- **연결 강도 시각화**: 그래프에서 연결 횟수/맥락 유사도를 링크 굵기로 표현
- **모바일 최적화**: 현재 데스크탑 기준 레이아웃
- **Vercel 배포**: 로컬 개발 완료 후 배포
